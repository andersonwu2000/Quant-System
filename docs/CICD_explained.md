# 持續整合與持續部署（CI/CD）完整解說

> 資料來源：Red Hat、Atlassian、GitLab、IBM、GitHub、JetBrains 等（2026-03-23）

---

## 一、核心概念：CI/CD 是什麼？

- **CI/CD** 是 **Continuous Integration（持續整合）** 與 **Continuous Delivery/Deployment（持續交付／持續部署）** 的縮寫
- 核心精神：**把軟體開發中反覆、容易出錯的手動步驟，全部自動化**
- 可以想像成一條**自動化生產線**：程式碼進去，經過層層品管檢測，最後自動上架到使用者面前
- 是現代 DevOps 文化的骨幹，也是大型科技公司（Netflix、Facebook、Google）能高速出貨的關鍵秘訣

---

## 二、三個核心概念的區別

### 持續整合（CI — Continuous Integration）
- 開發者每天多次把自己的程式碼**合併回主幹**（main branch）
- 每次合併後，系統自動執行**編譯 + 測試**
- 比喻：像是每天下班前把今天做的積木拼回大模型，馬上確認有沒有拼錯
- 目的：**儘早發現 Bug，避免問題積累到最後爆炸**

### 持續交付（CD — Continuous Delivery）
- 程式碼通過所有測試後，自動打包好、準備好可以上線
- **但最後按下「部署」按鈕，需要人工確認**
- 比喻：商品已經裝箱上貨車，但需要主管點頭才能出貨

### 持續部署（CD — Continuous Deployment）
- 比「持續交付」更激進：測試全過就**全自動上線，不需要人工介入**
- 比喻：商品生產完、品檢通過，直接自動送到顧客手上
- 適合成熟團隊，對自動化測試覆蓋率要求非常高

---

## 三、CI/CD 流水線（Pipeline）運作步驟

```
程式碼提交 → 觸發 → 建置 → 測試 → 部署
```

### 步驟一：原始碼觸發（Source）
- 開發者把程式碼 `git push` 上傳到 GitHub / GitLab 等平台
- 這個動作自動**觸發 CI/CD 流水線**開始運作

### 步驟二：建置（Build）
- 系統自動把程式碼**編譯**成可執行檔、函式庫或 Docker 容器映像檔
- 若此步驟失敗（如語法錯誤），馬上通知開發者，後續步驟停止

### 步驟三：測試（Test）—— 最重要的關卡
- **單元測試（Unit Test）**：測試每個小功能是否正確，速度最快、最先跑
- **整合測試（Integration Test）**：測試各模組合作時是否正常
- **功能測試（Functional Test）**：從使用者角度模擬操作，確認行為正確
- **安全測試（Security Scan）**：掃描是否有已知漏洞（現代工具會自動做這步）
- 任何一關失敗，**整條流水線暫停，通知相關人員**

### 步驟四：部署（Deploy）
- 所有測試通過後，將程式自動（或手動審核後）推上正式環境
- 進階做法：**金絲雀部署（Canary Deploy）**—— 先讓 5% 的使用者用新版本，確認沒問題再全面推出

## 四、為什麼企業要用 CI/CD？

- **加速發布**：從過去「每季更新一次」縮短到「每天甚至每小時更新」
- **降低風險**：每次只改一小部分，出問題容易找到原因；相比一次大更新更安全
- **減少人為失誤**：自動化流程取代手動操作，避免「忘記測試就上線」
- **節省成本**：Bug 在開發階段抓到，修復成本比上線後再修少 10 倍以上
- **快速回滾**：出問題時可以一鍵退回上一個穩定版本
- **提升團隊士氣**：工程師不再需要熬夜手動部署，也不怕週五下班前的更新

## 五、常見工具一覽

| 工具 | 特色 | 適合對象 |
|------|------|----------|
| **GitHub Actions** | 與 GitHub 深度整合，易上手，範本豐富 | 開源專案、中小型團隊 |
| **GitLab CI/CD** | 內建整個 DevOps 平台，安全掃描強 | 企業、需要自架的團隊 |
| **CircleCI** | 速度快，Docker 支援佳，被 Meta、Adobe 採用 | 需要高效能 CI 的團隊 |
| **Jenkins** | 最彈性、插件最多（1,900+），但設定複雜 | 有特殊需求的大型企業 |
| **AWS CodePipeline / Azure DevOps** | 與各自雲端生態系深度整合 | 已採用 AWS/Azure 的團隊 |

## 六、真實世界案例

### Netflix
- 採用**微服務架構**搭配 CI/CD，數百個獨立服務各自有自己的流水線
- 使用「混沌工程（Chaos Engineering）」：主動在正式環境製造故障，確保系統夠強健
- 金絲雀部署：新功能先推給少數用戶，收集回饋後再全面上線

### Facebook（Meta）
- 數千名工程師同時開發，每天有**數百次程式碼提交**
- 每天至少**部署兩次**到生產環境，依然保持穩定

---

## 七、最佳實踐（做 CI/CD 的正確姿勢）

- **測試要分層**：單元測試最快先跑，整合測試後跑，避免等待時間過長
- **同一份產出物貫穿所有環境**：不要在不同環境重新編譯，確保「測試的」就是「上線的」
- **環境要隔離**：每次測試都用全新乾淨的環境（Docker 容器），避免「在我電腦上沒問題」的情況
- **讓主幹永遠可發布**：程式碼只要進了主幹，必須保持隨時可以上線的狀態
- **快速失敗（Fail Fast）**：流水線一有問題立刻通知，不要等到全部跑完
- **逐步擴大自動化範圍**：不需要一步到位，先從 CI 開始，再逐漸加入 CD

> **CI/CD 就是替軟體開發打造一條自動化品管流水線：程式碼進去，層層把關，品質合格就自動送到使用者手上——既快又安全。**
