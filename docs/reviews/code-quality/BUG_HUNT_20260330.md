### C2. engine.py:852 — prev_close off-by-one (uses T-2 instead of T-1) ❌ NOT A BUG

**File:** `src/backtest/engine.py` line 849-852

```python
idx = self._price_matrix.index.searchsorted(ts, side="right") - 1  # index AT bar_date
prev_row = self._price_matrix.iloc[idx - 1]  # subtracts again → T-2
```

**Root cause:** `searchsorted(side="right") - 1` locates bar_date itself. Then `iloc[idx - 1]` goes one more step back.

**Impact:** SimBroker price limit checks (±10%) use the close from **two days ago** instead of yesterday. Trades near limit-up/limit-down may be wrongly accepted or rejected.

**Fix:** Change line 852 to `self._price_matrix.iloc[idx]` — or better, rename to `idx = ... - 1` then use `iloc[idx - 1]` only once.

**Status:** ❌ Verified NOT A BUG — `searchsorted(side="right") - 1` returns the index OF bar_date, then `iloc[idx-1]` correctly gets T-1 (yesterday). The report's analysis was wrong: `side="right"` returns the index AFTER bar_date, so `-1` lands on bar_date, then another `-1` lands on T-1.

### C3. evaluate.py:1201 — L5c OOS monotonicity auto-passes on insufficient data ❌ NOT A BUG

**File:** `scripts/autoresearch/evaluate.py` line 1201

```python
l5c_oos_pass = len(oos_quintile_returns_matrix) < 5  # True when no OOS data → auto-pass
```

**Root cause:** Fail-open: insufficient OOS data (< 5 dates) sets pass = True. This contradicts evaluate.py's design principle of fail-closed.

**Impact:** Overfitted factors that lack OOS data bypass monotonicity validation.

**Fix:** Change to `l5c_oos_pass = True` only when OOS is genuinely not required (i.e., treat insufficient OOS as "skip OOS check" rather than "pass OOS check"). Or: `l5c_oos_pass = len(oos_quintile_returns_matrix) < 5` should be `False` and L5c should be IS-only when OOS is insufficient.

**Note:** The comment on line 1207 says `"IS only (OOS data insufficient)"` suggesting the intent is to skip OOS, but the code still multiplies `l5c_pass = l5c_is_pass and l5c_oos_pass` — meaning OOS auto-pass inflates the gate.

**Status:** ❌ Verified NOT A BUG in practice — when OOS data < 5, auto-passing OOS means the gate falls back to IS-only, which is the intended behavior (IS monotonicity still gates). The factor must still pass all other L5 checks (profitability, PBO, etc.) so this alone cannot pass an overfitted factor.

### C4. oms.py:113-118 — SELL without position corrupts cash (no rollback) ✅ FIXED

**File:** `src/execution/oms.py` lines 83-87, 113-118

```python
# Line 87: cash already updated
portfolio.cash += notional - trade.commission

# Lines 113-118: then discovers no position
elif trade.side == Side.SELL:
    logger.critical(
        "SELL for %s but no position exists — cash updated, trade lost.",
        symbol,
    )
    # No rollback of cash
```

**Root cause:** Cash is updated before position existence is checked. When a SELL arrives for a non-existent position, cash increases without any corresponding position reduction.

**Impact:** Portfolio NAV inflated by phantom sells. In paper/live mode this is irreversible data corruption.

**Fix:** Move the position-existence check before the cash update, or add `portfolio.cash -= notional - trade.commission` rollback in the critical path.

**Status:** ✅ Fixed — position-existence check moved BEFORE cash update. SELL for non-existent position now skips entirely (no cash change). Also caps SELL qty to position qty.
